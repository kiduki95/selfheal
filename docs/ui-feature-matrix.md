# selfheal — UI 기준 기능 인벤토리 & 실시간성 매트릭스

> 버전: v0.1 (draft) · 대상 독자: 백엔드/UI 개발자
>
> 이 문서는 **`web/` UI가 그려놓은 화면을 기준으로**, 각 화면이 실제로 동작하려면 어떤 백엔드 기능이 필요한지, 그게 지금 있는지, 그리고 **그 기능이 실시간이어야 하는지 배치여도 되는지**를 정리한다. UI는 현재 전부 mock 데이터(`web/src/data/*.ts`)로 돌아간다. 레이어 정의·데이터 모델은 [architecture.md](./architecture.md) 참조.
>
> **한 줄 진단**: UI는 풀 루프(수집→처리→인사이트→승인→Auto-Dev→PR)를 다 그렸지만, 백엔드는 **가운데 3개 레이어(처리·인사이트·코드그래프)만 실제 동작**한다. 양끝(수집·Auto-Dev)과 운영 기능(활동로그·설정·인증·검색)은 비어 있다.

---

## 범례

**구현 상태**
- ✅ 구현됨 — 실데이터 서빙 가능
- 🟡 부분/미연결 — 코드는 있으나 API 미배선이거나 일부만
- 🔴 없음 — 테이블/코드 전무

**실시간성** — *"이 기능이 데이터를 어떤 박자로 다뤄야 하는가"*
- ⚡ **실시간** — 사용자가 보는 중에 상태가 바뀌어 **push가 필요**(SSE/WebSocket). 지연 목표 초 단위.
- 🔄 **준실시간** — **요청/응답 또는 짧은 폴링**. 화면 진입 시 최신값 fetch, 액션 시 즉시 응답.
- 🗓 **배치** — **스케줄 또는 수동 트리거**로 도는 잡. 분~시간 단위, 결과는 나중에 조회.

> 한 기능이 쓰기/읽기 박자가 다르면 **UI가 마주하는 쪽**을 기준으로 표기하고 비고에 보충한다. (예: 수집은 쓰기=🗓배치 폴링, 읽기=🔄준실시간 조회)

## 1. 수집 — Sources 페이지 · 대시보드 인제스션 히트맵

| 기능 | 구현 | 실시간성 | 비고 |
|---|---|---|---|
| 소스 커넥터(App/Play Store·Reddit·X·Intercom·경쟁사) | 🔴 | 🗓 배치 | 스토어 리뷰는 스트림이 아님 → N분 폴링. Intercom 등 webhook 소스는 ⚡push 가능 |
| 소스 등록·인증(Add source 오버레이) + OAuth/자격증명 저장 | 🔴 | 🔄 준실시간 | 요청/응답, 연결 테스트 |
| 소스 상태/주기/rate/lastSync (ok·warn·err) | 🔴 | 🔄 준실시간 | 화면 진입 시 조회, 30~60s 폴링이면 충분 |
| own(자사) vs competitor 구분 | 🔴 | 🔄 준실시간 | 소스 메타 |
| 인제스션 히트맵(소스 × 7일) | 🔴 | 🗓 배치 | 집계 산출물 |
| 자동 폴링/수집 스케줄 | 🔴 | 🗓 배치 | 스케줄러 자체가 없음(§5 참조) |

## 실시간성별 요약 — *진짜 실시간이 필요한 건 손에 꼽는다*

설계상 가장 중요한 결론: **이 제품의 거의 모든 화면은 요청/응답(🔄) + 배치(🗓) 조합으로 충분**하고, **스트리밍 인프라(SSE/WebSocket)가 실제로 필요한 곳은 소수**다. 비용·복잡도 큰 실시간 채널을 전 화면에 깔 필요 없음.

**⚡ 실시간(push) — 여기만 스트리밍 투자**
- Auto-Dev 터미널 로그 / 스텝 진행률 / PR 체크 (§4) — 핵심. SSE면 충분(단방향).
- 알림 벨 (§5) — 폴링으로 시작해도 무방.
- Slack 인터랙션 콜백 (§3) — Slack이 push해줌(우리가 받는 쪽).

**🗓 배치(스케줄/트리거) — 루프의 무거운 일들**
- 수집 폴링, 처리 파이프라인, 인사이트 생성/재생성, re-cluster, 대시보드 집계·스냅샷, 스케줄러.

**🔄 준실시간(요청/응답·폴링) — 나머지 전부**
- 모든 조회 화면(리뷰·그래프·제안·소스·활동·대시보드 read), 승인/반려, 설정, 검색, 필터, 인증.

## 빌드 우선순위 (architecture 로드맵 단계와 정렬)

**큰 구멍 3개 (순서대로)**
1. **활동/감사 로그 레이어** (roadmap §8) — 가장 싸고 효과 큼. `audit_events` 테이블 + 파이프라인/승인/스케줄 지점 로깅만으로 대시보드 activity 피드 + Activity 페이지 동시 활성화. 전부 🔄/🗓 — 실시간 인프라 불필요.
2. **수집 레이어** (§6) — `sources` 테이블 + 커넥터 1~2개(App Store/Reddit)부터. 대시보드 퍼널 1단계·히트맵·Sources 페이지가 실데이터화. 🗓 배치 폴링.
3. **Auto-Dev + GitHub** (§7) — 가장 무겁고 위험. 루프를 닫는 핵심이며 **이 제품에서 유일하게 ⚡실시간(SSE) 채널이 필요**한 부분이라 마지막.

**빠르게 메꿀 수 있는 것들 (테이블 추가 없이, 전부 🔄/🗓)**
- `impact.ts`/`lifecycle.ts` → `runInsight` 연결 (impactScore·effort 실값화)
- `/api/proposals/regenerate`, `/api/graph/recluster` 트리거 엔드포인트
- `/api/reviews` CSV export
- 대시보드 sparkData를 `metric_snapshots` 시계열로 채우기
- `/api/reviews` 필터/날짜범위 쿼리 파라미터 배선
